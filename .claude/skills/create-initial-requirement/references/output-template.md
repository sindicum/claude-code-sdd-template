## 記述ルール

### 書くべきこと（このファイルの責務）

- 何を作るか、なぜ作るか
- 機能の名前と1文の概要
- やらないことの明示
- 優先順位と実装順序
- 前提条件と制約
- 未決事項

### 書かないこと（後続工程の責務）

- 技術スタックの詳細（フレームワーク、ライブラリ名等）
- APIエンドポイント設計
- データベーステーブル定義
- 入力/出力の型定義
- テストケースの詳細
- UI のワイヤーフレームやコンポーネント設計

### 機能概要の粒度

各機能の「概要」は1文で書く。詳細仕様は後続工程で定義する。

```
OK: 「ユーザーが商品を検索し、カテゴリ・価格帯でフィルタできる」
NG: 「GET /api/products?category=xxx&min_price=100 で商品を取得し、
     レスポンスは JSON で { items: [...], total: number } を返す」
    → これは実装仕様であり、このファイルの責務外
```

### やらないことリストの書き方

理由を添えることで、後から「なぜやらないと決めたか」を追跡できるようにする。

```
OK: 「多言語対応: 初期ユーザーは国内のみのため」
NG: 「多言語対応」（理由なし → 後で判断根拠が不明になる）
```
